# ТЗ: Вкладка "Schema & Social Preview"

## 1. Назначение

Новая вкладка в расширении (далее — SERP Zen), отображающая два независимых блока анализа текущей страницы:

1. **Schema Markup** — структурированные данные (JSON-LD, Microdata, RDFa)
2. **Social Preview (Open Graph)** — мета-теги превью для соцсетей (Open Graph, Twitter Cards, Facebook-специфичные теги)

Оба блока живут на одной вкладке, друг под другом, по аналогии с существующей вкладкой "Index" (карточный список со статус-индикатором).

---

## 2. Общая архитектура

### 2.1 Извлечение данных (content script)

Весь сбор данных выполняется в content script (доступ к `document` текущей страницы), результат передаётся в panel/popup через `chrome.runtime.sendMessage`.

**Schema Markup:**

- JSON-LD: `document.querySelectorAll('script[type="application/ld+json"]')`, парсинг через `JSON.parse`, поддержка как одиночного объекта, так и массива объектов, и `@graph`-обёртки
- Microdata: рекурсивный обход `[itemscope]`, для каждого — `itemtype` + дочерние `[itemprop]` (не заходя внутрь вложенных `itemscope`)
- RDFa: поиск `[typeof]` + `[property]`

Все три формата нормализуются в единую структуру:

```ts
interface SchemaBlock {
  format: 'json-ld' | 'microdata' | 'rdfa';
  type: string; // например "Product", "Article", "Organization"
  properties: Record<string, any>;
  raw: string; // исходный код блока для отображения
  sourceIndex: number; // порядковый номер на странице
}
```

**Social Preview / OG:**

- Сбор всех `<meta property="...">` и `<meta name="...">`
- Дополнительно: `<title>`, `<link rel="canonical">`, `<link rel="icon">`, `<link rel="apple-touch-icon">`

```ts
interface MetaTag {
  key: string; // 'og:title', 'twitter:image', 'description'...
  value: string | null;
  group: 'og' | 'twitter' | 'facebook' | 'article' | 'basic';
}
```

### 2.2 Динамический контент

Для SPA (React/Vue и т.п.), где теги могут появиться после первичной загрузки:

- `MutationObserver` на `document.head`
- Повторный сбор данных при изменении набора `meta`/`script[type="application/ld+json"]`
- UI должен уметь обновляться "вживую" без необходимости перезапуска анализа пользователем

### 2.3 Проверка доступности изображений (валидация)

Для og:image / twitter:image нужно знать реальные размеры и доступность файла:

- Загрузка изображения через `new Image()` в фоне (popup/panel context, не content script, чтобы не блокировать страницу)
- Если URL относительный — резолвить относительно `location.href`
- Таймаут на загрузку — 5 секунд, после которого статус — "недоступно/таймаут"
- Сохранять: `{ url, naturalWidth, naturalHeight, fileSizeBytes (если доступно через HEAD-запрос), loadStatus: 'ok' | 'error' | 'timeout' }`

## 3. Блок 1: Schema Markup

### 3.1 UI-структура

Карточка-группа на каждый найденный schema-блок (по аналогии со скрином Index, но с возможностью разворачивания):

```
┌─────────────────────────────────────────────────────┐
│ 🟢 Product (JSON-LD)                      [Valid]   │  ← заголовок группы, кликабелен
├─────────────────────────────────────────────────────┤
│   🟢  name              "Кроссовки Nike Air"   OK    │
│   🔴  price              —                    Missing│
│   🟡  availability       "InStock"             Warning│ ← неизвестное значение enum
│   🟢  image               https://...          OK    │
└─────────────────────────────────────────────────────┘
```

- Заголовок группы: иконка-статус + `@type` + формат (JSON-LD/Microdata/RDFa) + общий статус-бейдж (`Valid` / `Has Errors` / `Has Warnings`)
- Внутри — построчно свойства в стиле существующих карточек Index (название свойства / значение / статус-бейдж)
- Если блоков schema на странице несколько — рендерятся последовательно, каждый в своей карточке
- Если schema не найдена вообще — единая карточка-заглушка: `🔴 Schema Markup не найден` + статус `Missing`

### 3.2 Источник правил валидации

Источник истины — официальная спецификация **schema.org** (актуальная версия, требуемые/рекомендуемые свойства зависят от `@type`) и требования **Google Rich Results** для тех типов, что поддерживают rich-сниппеты (Product, Article, Recipe, FAQ, Review, BreadcrumbList, Organization, LocalBusiness, Event, JobPosting и т.д.).

> ⚠️ Перед реализацией нужно явно зафиксировать список поддерживаемых на старте `@type` (см. раздел 6 "Открытые вопросы"), поскольку полный охват всех типов schema.org в MVP избыточен.

### 3.3 Логика валидации (полная, по спецификации)

Для каждого найденного `@type` система проверяет:

| Категория проверки                                    | Пример                                                                    | Статус при нарушении                                                                        |
| ----------------------------------------------------- | ------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------- |
| Обязательное свойство отсутствует                     | `Product` без `name`                                                      | 🔴 Missing                                                                                  |
| Обязательное свойство пустое/null                     | `"name": ""`                                                              | 🔴 Missing                                                                                  |
| Рекомендуемое свойство отсутствует                    | `Product` без `aggregateRating`                                           | 🟡 Warning                                                                                  |
| Неверный тип значения                                 | `price` — строка с буквами вместо числа                                   | 🔴 Invalid format                                                                           |
| Невалидный enum                                       | `availability: "MaybeInStock"` (не входит в допустимый список schema.org) | 🔴 Invalid value                                                                            |
| Невалидный URL в URL-поле                             | `image: "картинка.jpg"` без протокола/некорректный                        | 🔴 Invalid URL                                                                              |
| Невалидная дата                                       | `datePublished: "вчера"` (не ISO 8601)                                    | 🔴 Invalid format                                                                           |
| Синтаксическая ошибка JSON-LD                         | `JSON.parse` упал                                                         | 🔴 Critical — весь блок помечается как broken, показывается raw-текст с указанием на ошибку |
| Дублирующиеся обязательные блоки конфликтующего типа  | Два разных `Organization` с разными `name` на одной странице              | 🟡 Warning — "Conflicting data"                                                             |
| Вложенные структуры (`image: { @type: ImageObject }`) | Неполный вложенный объект                                                 | Рекурсивная проверка вложенного типа по тем же правилам                                     |

Каждая строка свойства внутри карточки получает один из 4 статусов:

- 🟢 **OK** — присутствует и валидно
- 🟡 **Warning** — рекомендуемое поле отсутствует, либо есть мягкое несоответствие
- 🔴 **Missing** — обязательное поле отсутствует/пусто
- 🔴 **Invalid** — поле присутствует, но значение не проходит проверку формата/типа/enum

### 3.4 Дополнительно

- Кнопка/иконка "Показать raw" на каждой карточке — разворачивает исходный JSON-LD/атрибуты как есть (с подсветкой синтаксиса)
- Кнопка "Скопировать" для raw-блока
- Ссылка-подсказка на соответствующую страницу schema.org для типа (опционально, для контекста — "что значит это поле")

## 4. Блок 2: Social Preview (Open Graph)

### 4.1 UI-структура — список тегов

Тот же карточный паттерн, что и в блоке Schema/Index:

```
🟢  og:title               "Заголовок страницы"          OK
🔴  og:image                —                            Missing
🟢  og:description         "Описание длиной 120 симв..."  OK
🟡  og:image:width          —                           Warning
🟢  twitter:card           "summary_large_image"          OK
🔴  twitter:image           —                            Missing
```

Группировка строк по подсекциям с подзаголовками внутри блока:

- **Open Graph** (`og:*`)
- **Twitter / X Cards** (`twitter:*`)
- **Facebook-specific** (`fb:app_id`, `fb:pages`)
- **Article meta** (`article:*`) — показывается только если присутствует хотя бы один тег этой группы
- **Базовые meta** (`title`, `description`, `canonical`) — для контекста, т.к. многие платформы используют их как fallback при отсутствии OG-тегов

### 4.2 Визуальная имитация карточки (Link Preview Simulator)

Над/под списком тегов — переключаемые вкладки-имитации того, как ссылка выглядит при шаринге в разных сетях:

```
[ Facebook ]  [ X (Twitter) ]  [ Telegram ]  [ LinkedIn* ]

┌──────────────────────────────────┐
│                                   │
│         [ og:image ]             │
│                                   │
├──────────────────────────────────┤
│ DOMAIN.COM                       │
│ Заголовок (og:title)             │
│ Описание (og:description)        │
└──────────────────────────────────┘
```

Особенности рендера на каждой "вкладке-сети" (визуально имитируем характерные отличия):

| Платформа       | Источник title/desc/image                 | Особенности отображения                                                                                                |
| --------------- | ----------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| **Facebook**    | `og:*`, fallback на `title`/`description` | Картинка сверху, домен капсом снизу заголовка                                                                          |
| **X (Twitter)** | `twitter:*`, fallback на `og:*`           | Зависит от `twitter:card`: `summary` — картинка слева маленькая, `summary_large_image` — картинка во всю ширину сверху |
| **Telegram**    | `og:*`                                    | Похоже на Facebook, но без домена под заголовком, текст описания обычно длиннее видимой части                          |
| **LinkedIn**    | `og:*`                                    | Похоже на Facebook-вариант                                                                                             |

> Это упрощённая визуальная имитация (приближение), не точный рендер — платформы могут менять алгоритм кэширования/обрезки в любой момент. В UI должна быть пометка типа "примерная имитация" / disclaimer.

Если `og:image` не загрузилась/отсутствует — в карточке-имитации показывается серый placeholder с иконкой "битого изображения" + текст "Изображение не найдено".

### 4.3 Логика валидации (полная, по спецификациям)

| Тег                                  | Проверка                                                                  | Статус при нарушении                                                       |
| ------------------------------------ | ------------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| `og:title`                           | Отсутствует                                                               | 🔴 Missing                                                                 |
| `og:title`                           | Длина > 60-70 символов (будет обрезано в большинстве соцсетей)            | 🟡 Warning — "Будет обрезано"                                              |
| `og:description`                     | Отсутствует                                                               | 🔴 Missing                                                                 |
| `og:description`                     | Длина > 200 символов                                                      | 🟡 Warning — "Будет обрезано"                                              |
| `og:description`                     | Длина < 50 символов                                                       | 🟡 Warning — "Слишком короткое описание"                                   |
| `og:image`                           | Отсутствует                                                               | 🔴 Missing                                                                 |
| `og:image`                           | URL не загружается (404 / таймаут / CORS)                                 | 🔴 Invalid — "Изображение недоступно"                                      |
| `og:image`                           | Реальный размер < 200×200px                                               | 🔴 Invalid — Facebook минимум для краулинга                                |
| `og:image`                           | Реальный размер < 600×315px                                               | 🟡 Warning — "Ниже рекомендуемого для large preview"                       |
| `og:image`                           | Соотношение сторон далеко от 1.91:1                                       | 🟡 Warning — "Может обрезаться нестандартно"                               |
| `og:image`                           | Вес файла > 8 МБ (лимит Facebook)                                         | 🔴 Invalid — "Превышен лимит размера файла"                                |
| `og:image:width` / `og:image:height` | Указаны, но не совпадают с реальными размерами файла                      | 🔴 Invalid — "Заявленный размер не совпадает с фактическим"                |
| `og:url`                             | Отсутствует                                                               | 🟡 Warning                                                                 |
| `og:url`                             | Не совпадает с canonical/текущим URL                                      | 🟡 Warning — "Mismatch с canonical"                                        |
| `og:type`                            | Отсутствует                                                               | 🟡 Warning (по умолчанию считается `website`)                              |
| `og:type`                            | Значение не входит в список валидных типов OG                             | 🔴 Invalid value                                                           |
| `twitter:card`                       | Отсутствует                                                               | 🟡 Warning — будет use fallback на OG, но не оптимально                    |
| `twitter:card`                       | Значение не из списка (`summary`, `summary_large_image`, `app`, `player`) | 🔴 Invalid value                                                           |
| `twitter:image`                      | Отсутствует, но `og:image` есть                                           | 🟢 OK (fallback работает) — информационно показать "используется og:image" |
| `twitter:image`                      | Указан, но недоступен                                                     | 🔴 Invalid — "Изображение недоступно"                                      |
| `twitter:site` / `twitter:creator`   | Отсутствует, не начинается с `@`                                          | 🟡 Warning — формат                                                        |
| Дублирующиеся meta-теги              | Два `og:title` с разными значениями                                       | 🔴 Invalid — "Конфликтующие дубликаты"                                     |

### 4.4 Сводный статус блока

В заголовке блока "Social Preview" — общий бейдж:

- 🟢 **Ready to share** — нет критичных ошибок
- 🟡 **Needs improvement** — есть warnings, но критичных Missing/Invalid нет
- 🔴 **Broken preview** — есть хотя бы одна критичная ошибка (особенно отсутствие/недоступность og:image, отсутствие og:title)

## 5. Общие UI-требования (для обоих блоков)

1. Стиль карточек — переиспользовать существующий паттерн вкладки Index: белая карточка на сером фоне, круглый цветной индикатор слева, название/значение посередине, статус-бейдж справа
2. Цвета статусов:
   - 🟢 зелёный — OK
   - 🟡 жёлтый/оранжевый — Warning
   - 🔴 красный — Missing / Invalid / Critical
3. Длинные значения (URL, JSON) — обрезаются с `...` и полностью показываются по hover/клику (tooltip или раскрытие)
4. Состояние загрузки: пока идёт проверка og:image (запрос изображения, HEAD-запрос на размер файла) — показывать skeleton/spinner на соответствующей строке, не блокируя остальной UI
5. Кнопка "Обновить анализ" — на случай динамического контента (SPA), чтобы пересобрать данные без перезагрузки страницы
6. Состояние "страница без HTML head доступа" (например, `chrome://` страницы, PDF-вьюер) — отдельный пустой стейт с понятным сообщением, а не пустые карточки с ошибками

## 6. Открытые вопросы (требуют решения до разработки)

1. **Список поддерживаемых `@type` для Schema-валидации в MVP** — нужен ли полный охват schema.org (сотни типов) или фиксированный список приоритетных (Product, Article, Organization, LocalBusiness, FAQPage, BreadcrumbList, Review, Event, Recipe, JobPosting, VideoObject)?
2. **HEAD-запрос для получения веса файла og:image** — может блокироваться CORS на некоторых доменах; нужен ли fallback на оценку "недоступно для проверки" вместо ошибки?
3. **Источник правил валидации** — хардкодить правила в коде расширения, или вынести в обновляемый JSON-конфиг (чтобы обновлять лимиты соцсетей без релиза расширения)?
4. **LinkedIn-вкладка в Link Preview Simulator** — добавлять ли в MVP, учитывая что у LinkedIn нет полностью документированной спецификации превью (помечена `*` в разделе 4.2)?
5. **Хранение результатов между открытием/закрытием popup** — нужно ли кэшировать результат анализа на вкладку (`chrome.storage.session`), чтобы не пересчитывать при каждом открытии расширения?
